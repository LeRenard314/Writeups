## Эксплуатация

Попробуем получить доступ к панели администратора.

Для этого можно либо сразу попробовать обратиться к `/admin`, поскольку это распространенное название эндпойнта содержащего интерфейс администратора, либо профаззить приложение, однозначно установив, какие пути существуют, и какой из них отвечает за админскую панель:

``` Shell
ffuf -w /usr/share/dirb/wordlists/common.txt -u https://0ad3006303b8139c80362be500cb00a8.web-security-academy.net/FUZZ
```

``` Shell

Admin                   [Status: 401, Size: 2696, Words: 1060, Lines: 58, Duration: 754ms]
```

Теперь имеет смысл послать запрос на найденную панель.

Сервер возвращает `403`,  так просто на неё не зайти:

![placeholder](images/labAdmin.png)

Попробуем детальнее изучить запрос к эндпойнту. Описание лабы хинтит, что необходимо обнаружить скрытый хэдер, передаваемый при запросе.

Для этого пробуем послать `TRACE` запрос, по итогу обнаружим хэдер `X-Custom-IP-Authorization`. Вероятно, на стороне приложения крутится реверс-прокси, которая передает на бэкенд айпи адрес, с которого идет обращение, заворачивая его в найденный хэдер:

![placeholder](images/labTraceAdminCensored.png)

В данном случае, нам повезло - метод `TRACE` разрешен. Если бы воспользоваться им было невозможно, на помощь приходит фаззинг заголовков. Конкретно по этому кейсу - заголовок `X-Custom-IP-Authorization` дернуть вне контекста почти невозможно, т.к он выдуман командой `Portswigger` и вряд ли встретится в боевых ситуациях (и соответственно в вордлистах тоже), как они сами и признаются. Но я его все равно сохранил в  словарик, на всякий пожарный^^

Идейно, перебор выполнялся бы так (мы догадываемся, какое значение может принять искомый заголовок):

``` Shell
ffuf -w headers-fuzz.txt -u https://0a94000204dc7fe7803f3a9600be004e.web-security-academy.net/admin -H "FUZZ: 127.0.0.1" -mc 200
```

Если же мы без понятия, какой использован заголовок или какое значение с помощью него передается, можно попробовать подцепить его техникой диференциального перебора (`-mode clusterbomb`).

Чтобы обойти аутентификацию, можно попробовать заспуфить хэдер посылаемый на запрос. Для этого нужно поменять значение IP на `localhost`/`127.0.0.1`.

## Противодействие

Основная уязвимость кроется в возможности  произвести сруфинг заголовка `X-Custom-IP-Authorization`, обнаружение которого стало возможным ввиду того что разрешен метод `TRACE`.

Сервер/реверс-прокси должны проверять содержимое заголовка, отсеивая реквесты с признаками спуфинга, а запросы от клиента должны поддерживать только `http` методы как `GET`, `POST`, `OPTIONS`.
